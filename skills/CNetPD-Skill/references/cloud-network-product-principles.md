# 云网络产品顶层设计原则

这些原则用于复杂 PRD 方案评审、产品选型、需求分析、跨云差距判断和架构取舍。它们是思考和决策的纲领，不是回答模板。简单 API 查询、字段解释、命令定位、事实核验类问题无需完整套用。

对用户输出时，不要说“符合/违反某条原则”或逐条背诵本文。只在解释产品判断时，用自然语言说明基于哪些取舍：客户业务创新、技术成熟度、云原生/AI Native 范式、安全高可用、长期可复用性、成本和运维复杂度。

## 1. 避免网络限制客户业务创新

云网络的使命是避免网络能力边界、性能瓶颈、体验缺陷或交付复杂度限制客户业务创新，而不是无限制承接客户提出的一切网络诉求。

“不限制客户”不是无条件满足定制功能，而是不阻碍客户的业务架构演进、产品创新、规模扩张和商业试验。客户诉求超出现有能力时，先识别背后的业务约束，再判断是产品化建设、架构替代、能力增强，还是明确边界。

判断重点：

- 诉求是否服务于真实业务创新、业务连续性、规模增长或效率提升。
- 网络能力是否正在限制业务目标，而不只是客户希望获得某个定制功能。
- 方案是否能抽象为平台能力，而不是单一客户不可持续的特殊满足。
- 是否先识别业务约束，再决定产品化、替代方案、增强能力或边界说明。

## 2. 创新建立在成熟、稳定、可验证、可规模化的技术基础上

云网络产品创新必须建立在成熟、稳定、可验证、可规模化的技术基础之上。缺乏成熟底层技术、标准协议或可靠工程实践支撑时，网络技术创新应保持克制。

小规模验证可行，不代表适合作为正式云网络产品能力。短期解决局部问题的技术，如果引入新的协议栈、控制逻辑、运维范式或故障模式，长期可能带来更高复杂度、稳定性风险和排障成本。

判断重点：

- 是否有成熟底层技术、标准协议或可靠工程实践支撑。
- 是否为了局部诉求引入新的架构形态、协议栈、控制逻辑或运维范式。
- 新复杂性是否有明确业务价值和规模化收益。
- 规划规模下的数据面性能、控制面压力、稳定性风险和故障半径是否可控。
- 单位资源成本、长期维护成本、排障成本和演进成本是否可接受。
- 能力是否能沉淀为可持续产品能力，而不是长期不可维护的特殊实现。

## 3. 面向云原生和 AI Native 场景进行范式级创新

云网络产品创新不能局限于复刻传统物理网络能力，应面向云原生和 AI Native 场景进行范式级创新，通过产品抽象暴露业务意图，而不是把底层网络复杂性转嫁给客户。

云网络面对多租户、弹性调度、自动化交付、软件定义、安全隔离、跨地域互联和业务快速变化的环境，需要在网络抽象、资源编排、交付方式、可观测性、自动化运维和体验闭环上形成新的产品范式。

产品应尽量让客户表达业务意图，例如谁可以访问谁、需要什么隔离等级、性能保障和高可用目标，而不是要求客户理解过多底层网络细节、协议机制、转发路径和平台内部实现。

面向 AI Native 场景，还要关注大规模分布式训练、推理集群、东西向高吞吐通信、任务级调度、算网协同、确定性性能和智能化运维。

判断重点：

- 是否只是复刻传统物理网络能力，还是解决云场景下新的资源组织、交付和运维问题。
- 是否面向弹性、自动化、多租户、可编排和平台化需求。
- 是否通过产品抽象暴露业务意图，而不是转嫁底层复杂性。
- 是否降低客户理解成本、配置成本、接入成本和排障成本。
- 是否支撑 AI Native 场景下的高性能、确定性、自动化和算网协同。
- 是否在成熟度、成本、安全和高可用边界内形成云网络自身产品范式。

## 4. 安全与高可用是一切功能特性的前提

安全与高可用是云网络一切功能特性的前提。任何网络能力建设都不得以削弱安全边界、租户隔离、系统稳定性和故障恢复能力为代价。

云网络承载客户核心业务流量，安全和高可用不是附加要求，而是功能成立的前提。需求、产品创新或架构演进都必须先满足安全隔离、权限控制、故障隔离、可恢复性和稳定运行，再讨论功能完整性、性能优化和体验提升。

架构上优先追求静态稳定性和单元化。静态稳定性要求依赖异常、控制面故障、局部组件失效或自动化流程中断时，核心网络能力仍尽量依靠已有配置、已有转发表、已有状态和预置策略维持。单元化要求边界清晰、故障半径可控、可独立部署、可独立恢复、可横向复制。

判断重点：

- 是否削弱租户隔离、安全边界、权限控制或访问控制。
- 是否降低核心链路、控制面、数据面或运维面的高可用能力。
- 控制面异常、依赖服务异常或局部组件故障时，核心网络能力是否仍能维持。
- 故障是否能限制在单元内部，避免跨地域、跨集群、跨租户级联扩散。
- 架构是否简单、清晰、可复制、可灰度、可回滚、可恢复。
- 规模扩大后，系统复杂度、故障半径和运维成本是否仍可控。

## 5. 坚持长期主义，围绕十年不变的客户价值持续创新

云网络建设应坚持长期主义，围绕十年不变的客户价值持续创新，而不是被短期诉求、技术热点和阶段性竞争牵引。

业务形态、技术栈、客户规模和流量模型会变化，但连接、安全、隔离、稳定、性能、成本效率、自动化、可观测和易用性是长期不变的客户价值。创新资源应优先投入这些长期确定方向，而不是围绕短期定制、局部特例或未验证热点反复摇摆。

长期主义不是保守，而是把创新资源投入长期复利最大的方向。能持续降低客户网络复杂度、提升业务连续性、增强性能确定性、提高交付效率、降低单位成本、提升平台稳定性的能力，才值得长期建设。

判断重点：

- 是否面向客户长期持续关心的问题：安全、稳定、性能、成本、易用、自动化和可观测。
- 是否能形成长期可复用、可演进、可规模化的产品能力。
- 客户规模、业务形态和技术栈变化后是否仍有价值。
- 是否避免为了短期交付引入长期复杂性、技术债和不可维护的特殊实现。
- 是否能长期持续降低客户成本、平台成本和运维成本。
- 是否符合云网络长期架构方向，而不是只服务阶段性竞争或单点客户诉求。
