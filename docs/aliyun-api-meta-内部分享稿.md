# 从“会写”到“写得能用”：怎么让 AI 真正理解现有产品能力

这段时间，大家都开始用 AI 写 PRD、写方案、写文档。

这件事的上半场其实已经比较清楚了。让 AI 帮忙整理结构、润色表达、补齐章节，门槛已经不高。真正拉开差距的，反而不是“AI 会不会写”，而是“AI 知不知道我们现在已经有什么”。

因为我们的 PRD 设计，绝大多数都不是从白纸开始作画，而是在现有系统、现有产品、现有能力边界上做扩展。很多需求看起来是新需求，落到设计层面，本质上都是在回答几个老问题：

- 现有产品里有没有接近的能力
- 这个场景应该基于哪些已有组件组合出来
- 哪些能力已经支持，哪些还需要新做
- 如果要落到接口和资源层，应该往哪个产品、哪个 API 继续下钻

如果 AI 不知道这些，它写出来的内容就很容易停留在“像一份 PRD”的层面。结构看着完整，文字看着流畅，但方案不贴着现状走，产品边界不准，能力组合不准，最后只能拿来演示，拿不去落地。

所以，想让 AI 生成的 PRD 和文档从“能看”走到“真正可用”，有一个很关键的问题必须先解决：

**如何让 AI 准确理解现有的产品能力。**

## 为什么这件事很关键

AI 在公开知识上很强，但在企业内部设计场景里，光靠公开知识远远不够。

因为我们要的不是“阿里云网络大概有哪些产品”，而是更具体的问题：

- 做公网访问，这个场景在现有产品体系里该优先看什么
- 做跨地域互联，是 CEN、GA、VPC，还是几个产品一起上
- 做混合云接入，应该从 VPN、专线、ECR 哪条路径切
- 做一个新方案时，哪些能力其实已经有现成 API 和资源模型

这些问题如果没有一个足够贴近产品实际、足够可检索、足够可下钻的能力底座，AI 就只能自己猜，或者去互联网上搜公开文档。

问题也就来了。

## 只让 AI 去搜互联网公开文档，问题在哪

让 AI 自己去互联网搜文档，当然能解决一部分问题，但这条路有几个天然短板。

第一，文档新鲜度不稳定。公开搜索拿到的结果，未必是当前版本，未必是我们真正依赖的那份元数据，也未必能保证不同轮次问出来的内容一致。

第二，粒度经常不对。公开文档更适合人工浏览，不一定适合模型逐层定位。很多时候模型能搜到产品介绍，但不容易顺着场景一路找到具体能力分组，再下钻到 API 契约。

第三，跨产品关系弱。公开文档通常是按产品拆开的，适合查单点，不适合回答“这个场景该怎么组合现有能力”。

第四，搜索成本高。每次都让 AI 从外部搜索、筛选、判断、拼装，链路长、噪音多，稳定性也差。

第五，不够贴近我们的使用方式。我们内部讨论需求时，通常不是按“某个产品的全部文档”来思考，而是按“我要做公网访问”“我要做内网互通”“我要做负载均衡选型”这种场景来思考。

也就是说，公开文档不是不能用，而是它天然不是为“AI 协作式设计”这个场景组织的。

## 所以我们需要的，不是更多文档，而是一层 AI 可直接使用的能力组织

基于这个问题，我们做了一个更偏实用的尝试：`aliyun-network-api-skills`。

可以把它理解成一层给 AI 用的网络产品能力底座。它不是再造一份文档，而是把现有阿里云网络产品相关的 OpenAPI 元数据，重新整理成 AI 更容易理解和检索的结构，让模型在写 PRD、写方案、写说明文档时，先知道“已有能力是什么”，再去生成内容。

一句话说，它解决的是：

**不是让 AI 更会写，而是让 AI 写之前先知道该基于什么写。**

## `aliyun-network-api-skills` 里包含什么

这个 skill 当前覆盖的是阿里云网络域 13 个核心产品，不含 CDN 和 DNS，主要包括：

- 专有网络 VPC
- 云企业网 CEN
- 全球加速 GA
- 传统型负载均衡 CLB
- 应用型负载均衡 ALB
- 网络型负载均衡 NLB
- 网关型负载均衡 GWLB
- 网络智能服务 NIS
- 私网连接 PrivateLink
- 高速通道路由器 ECR
- VPC IP 地址管理 IPAM
- 任播 EIP
- VPC 对等连接

它不只是把这些产品平铺给 AI，而是做了两层组织。

第一层是产品层。AI 可以按产品查看能力分区、分组摘要、具体 API 详情，逐层下钻。

第二层是场景层。我们把这些产品重新整理成 7 个更符合设计思路的业务主题：

- 公网访问
- 负载均衡选型
- 跨地域互联
- 混合云接入
- VPC 间 / 服务间私网打通
- IP 地址管理
- 网络可视化与诊断

这很重要。因为真实的 PRD 讨论，大多数都不是从“我要查某个 API”开始，而是从“我要解决一个什么网络场景”开始。先按主题定位，再按产品下钻，AI 的理解路径会更接近实际设计过程。

除了主题和产品索引之外，这个 skill 里还带了：

- 每个产品的能力分组索引
- 每组下的 API 摘要
- API 级别的详细契约信息
- 查询脚本，方便 Agent 或模型按层检索
- 一些基于现有元数据推断出来的能力依赖线索

所以它不是一份静态资料包，更像一个可以被 AI 逐层访问的能力目录。

## 它的实现原理是什么

实现原理并不玄，核心就三步。

第一步，把阿里云官方 API Meta 数据拉下来，沉淀成一份本地、稳定、可控的原始数据源。

第二步，把原始单体文档拆成更适合模型读取的渐进披露结构。简单理解，就是不要一上来把整本说明书塞给 AI，而是先给目录，再给分组，再给详情。

当前这套结构大致是：

- L0：产品索引，告诉 AI 这个产品有哪些能力分区
- L1：分组概览，告诉 AI 这一组里有哪些 API、核心参数和关键返回
- L2：API 详情，只有在真正需要时才读取完整契约

第三步，在单产品结构之上，再做一层网络域重组。也就是不只保留“按产品查”的路径，还补上“按场景查”的路径，把多个产品编织成一个可以支持方案推理的网络能力地图。

这套思路背后的核心判断很简单：

**AI 不是看不到文档，AI 是看不到正确组织过的文档。**

## 这个 skill 可以实现什么

如果把它放到日常协作里看，它至少能做几件很实际的事。

第一，帮助 AI 写更贴近现状的 PRD。
AI 在写方案前，先知道现有网络产品能做什么、边界在哪里、常见场景怎么选型，生成的内容就不容易飘。

第二，帮助 AI 写更靠谱的设计说明和技术文档。
很多文档不是从零发明概念，而是解释“为什么这次选 ALB 而不是 NLB”“为什么这里走 CEN 而不是 VPC Peer”。有了现成能力地图，AI 的说明会更像基于现状推导，而不是基于公开常识拼接。

第三，支持从场景直接回到产品和 API。
比如先从“跨地域互联”这个主题进入，再定位到 CEN、GA、VPC 等产品，最后再继续下钻到具体能力和接口。

第四，减少人在上下文补课上的成本。
很多时候我们不是不会设计，而是每次都要先把现有能力重新讲给 AI 听。有了 skill，这部分重复劳动可以明显减少。

第五，让 AI 协作更稳定。
同样的问题，如果每次都靠自由搜索，AI 的输出会受搜索结果、排序、抓到的页面版本影响。有了本地化、结构化、可重复访问的能力包，结果会稳定很多。

## 相比让 AI 去搜公开文档，它的优势是什么

核心优势不是“信息更多”，而是“信息更适合被协作使用”。

相比公开搜索，这套方式可以规避或者明显缓解下面这些问题：

- 文档新鲜度问题：数据源可以统一更新，不依赖搜索时命中的外部页面
- 文档粒度问题：不是只有产品介绍，而是能一路下钻到能力分组和 API 详情
- 跨产品拼装问题：主题层把多产品能力重新组织起来，更适合方案设计
- 上下文噪音问题：AI 不需要每次重新扫大量无关页面
- 输出稳定性问题：相同问题基于同一套本地 skill，回答更一致
- 协作效率问题：减少“先找资料、再筛资料、再拼资料”的重复过程

换句话说，它把“AI 自己去外面找资料”变成了“AI 直接基于一套我们整理好的能力底座工作”。

这件事的意义在于，AI 协作终于不只是语言层的协作，而开始具备了产品能力层的协作基础。

## 这件事的目标，不只是提升回答质量

如果只是为了让回答看起来更专业，其实还不值得专门做一层 skill。

真正有价值的是，它把 AI 的角色往前推了一步。

以前 AI 更像一个文字助手，擅长改写、总结、润色。

有了这类能力底座之后，AI 才有机会变成一个更像“熟悉现有产品盘子”的协作对象。它不只是帮你把话说顺，而是开始理解：

- 现有能力大概长什么样
- 一个场景通常会落到哪些产品组合
- 方案讨论应该先从哪里切入
- 什么时候该继续下钻到接口级信息

这也是为什么我会觉得，`aliyun-network-api-skills` 的价值不只是一份网络资料包，而是一种更高效的 AI 协作方式。

## 最后

大家都在用 AI 写东西，这件事本身已经不是问题。

真正的问题是，怎么让 AI 写出来的东西不只是像样，而是真的能接上我们的现有系统、现有产品和现有能力。

`aliyun-network-api-skills` 想解决的，就是这个中间缺口。

它把原本分散在公开文档和产品资料里的网络能力，整理成一套 AI 可以稳定读取、逐层下钻、按场景组织的能力底座。这样一来，AI 生成 PRD 和文档时，不再只是基于泛化知识做“合理想象”，而是能更多地基于现有能力做“约束内设计”。

从这个角度看，这件事本质上不是在做文档整理，而是在给 AI 协作补产品上下文。

而一旦这层上下文补上，AI 才有可能真正从“演示用途”走向“生产可用”。
